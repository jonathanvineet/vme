# Phase 10.1 — Geometry Recovery Redesign

Implements the architectural fix for the 10B/10C/10F findings in `docs/audits/phase10/10.0_reconstruction_audit.md`: reconstruction was synthesizing straight lines from `family.length`/`orientation` instead of recovering the actual recognized shape. This replaces that synthesis with real geometry recovery, per your redesign.

## What changed

**New module `core/reconstruction/geometry_recovery.py`** — `recover_bar_path(component, graph, entity_by_geom_id)` walks the representative component's own connectivity-graph edges (not the family's aggregate stats) and returns an ordered polyline:
- **Simple path** (2 leaves, e.g. `straight_bar`, `l_bar`, `u_bar`): walks leaf-to-leaf in geometry order.
- **Closed loop** (0 leaves, all degree 2, e.g. `stirrup`): walks the full cycle.
- **Branch** (a degree-3+ node exists, e.g. `branch`): no single unambiguous bar path exists by definition. Recovers the longest leaf-to-leaf path by total edge length and **honestly records** `truncated_branch=True` + `excluded_edge_count`, rather than silently picking one arm and presenting it as the whole shape.
- ARC edges are sampled into `ARC_SAMPLE_SEGMENTS` (8) points using the entity's actual center/radius/start-end angles, so curvature is preserved as real geometry, not summarized into two endpoints.

**`PhysicalBarBuilder` rewritten**: recovers the representative's path once per family, then expands per-member copies (Phase 10C) by translating that recovered path along the family's own perpendicular axis by `member.offset_from_representative` — the same signed projection axis Phase 9 already uses, so a member at offset X ends up exactly X away from the representative while keeping every recovered bend/curve point, instead of re-deriving a fresh straight line per member.

**Diameter fabrication removed, per your instruction**: `PhysicalBar` gained `diameter_source` (`'annotation'` | `'missing_visual_fallback'`) and `diameter_confidence` (1.0 or 0.0). The `12.0mm` fallback still exists (a mesh needs *some* radius to sweep), but is now explicitly flagged rather than presented identically to real annotation data.

**`run_phase10.py` restructured**: previously read a frozen `engineering_families.json` snapshot with no path back to live geometry. Now rebuilds Phases 1-9 live (`_build_pipeline()`, mirroring `run_phase9.py`), matching every other `run_phaseN.py` script's convention, and giving `PhysicalBarBuilder` real access to `graph`/`comp_repo`/canonical entities. Since the full pipeline through Phase 9 is already proven deterministic, this reproduces byte-identical family data to whatever was last frozen — no risk of divergence, and it eliminates the lossy JSON round-trip (`_namespace()`/`asdict()` workarounds) entirely.

## Verified results

**Visual proof, not just numbers** (`debug/phase10/<drawing>/centerlines.png`): the N4 family, previously a flat straight line, now renders as a real **L-shaped bend**.

**Per-bar geometry, before vs after**:
| Mark | Shape | Before (points) | After (points) | Method | Truncated |
|---|---|---:|---:|---|---|
| N4 | l_bar | 2 (straight) | **10** | `simple_path` | No |
| N6 | branch | 2 (straight) | **58** | `longest_path_in_branch` | **Yes, 1 edge excluded** |
| N7 | branch | 2 (straight) | **20** | `longest_path_in_branch` | **Yes, 1 edge excluded** |

N4's 10 points match hand-verification exactly: 1 arc entity sampled into 9 points (8 segments + start) + 1 line endpoint = 10, confirming the sampling logic is doing exactly what it's documented to do.

**Diameter honesty**: N4 now correctly shows `diameter_source: annotation, confidence: 1.0` (it has real Phase 8 diameter data). N6/N7 now correctly show `diameter_source: missing_visual_fallback, confidence: 0.0` — previously indistinguishable from real data.

**Mesh scale**: vertex/face counts jumped from 156/288 to **2912/5376**, consistent with real multi-segment geometry now being swept (verified the math: 58-point N6 path → 57 segments × 26 verts/48 faces = 1482 verts/2736 faces alone, matching the total exactly).

**The predicted latent triangulator issue is now actively exercised, not fixed** (per your explicit instruction not to prematurely patch it): `BarTriangulator` still caps every segment independently, so N6's 57-segment sweep now produces 57 pairs of end caps at internal joints instead of one continuous tube. The mesh still reports `Watertight: True` (each segment is independently a valid closed cylinder, so the edge-count check doesn't catch this), but it's not a smooth continuous bar — this is exactly the documented, deferred "Phase 10E — Tube Sweep" task, now with a concrete, measurable example to build against instead of a hypothetical one.
