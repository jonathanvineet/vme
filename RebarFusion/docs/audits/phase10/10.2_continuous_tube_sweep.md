# Phase 10.2 — Continuous Tube Sweep: Design

Design-first, per instruction, before touching `triangulator.py`. Replaces the current per-segment independently-capped-cylinder approach (`docs/audits/phase10/10.0_reconstruction_audit.md`'s documented latent issue, now actively exercised per `10.1`) with a single continuous manifold tube per bar.

## Problem restated
`BarTriangulator.triangulate()` calls `_cylinder_segment(start, end, radius)` once per consecutive point pair and concatenates the results. Each call independently caps both its ends. For a `BarPath` with N points (N-1 segments), this produces N-1 separate closed cylinders that merely touch at shared vertices — not one continuous tube. Concretely, N6's 58-point path currently produces 57 pairs of end caps, almost all of which are redundant internal geometry that should not exist in a correctly swept tube.

## Target pipeline
```
BarPath.points
      ↓
Stage 1: Curve sampling      (respect existing point density; no resampling needed —
                               ARC edges are already adaptively sampled in geometry_recovery.py)
      ↓
Stage 2: Tangents             (per-sample direction, forward-difference at ends, central at interior)
      ↓
Stage 3: Parallel transport frames  (one (tangent, normal, binormal) triple per sample, no Frenet twist)
      ↓
Stage 4: Rings                 (N_SIDES vertices per sample, swept around the frame's normal/binormal)
      ↓
Stage 5: Bridge rings           (quad strip between consecutive rings, split into 2 triangles each)
      ↓
Stage 6: End caps               (exactly 2 total per bar: start and end, not per-segment)
      ↓
Stage 7: Normals                (per-vertex, averaged from adjacent face normals)
      ↓
Mesh (vertices, faces)
```

## Stage detail

### Stage 1 — Curve sampling
No new resampling stage is needed. `geometry_recovery.py` already samples ARC edges into `ARC_SAMPLE_SEGMENTS` points and passes LINE/POLYLINE_SEGMENT edges through as their two graph-node endpoints. The tube sweep consumes `BarPath.points` as-is. (A future refinement could add curvature-adaptive resampling if arc segment counts prove too coarse/fine in practice — not needed to fix the manifold-joint problem, so deferred.)

### Stage 2 — Tangents
For point `P_i`:
- Interior (`0 < i < N-1`): central difference, `tangent_i = normalize(P_{i+1} - P_{i-1})`.
- Start (`i=0`): forward difference, `normalize(P_1 - P_0)`.
- End (`i=N-1`): backward difference, `normalize(P_{N-1} - P_{N-2})`.
- Degenerate case (zero-length adjacent segment, i.e. `P_i == P_{i+1}`): reuse the previous valid tangent rather than producing a NaN/zero vector. `geometry_recovery.py` shouldn't emit duplicate consecutive points in practice (each edge contributes new points), but this is a defensive guard, not an assumption of clean input.

### Stage 3 — Parallel transport frames (not Frenet)
Frenet frames are rejected per your instruction: they're undefined/unstable wherever curvature is near zero (straight or near-straight segments — the majority of this project's geometry, given most recovered paths are one LINE plus a short ARC), causing visible twist artifacts. Parallel transport frames avoid this:

1. Seed frame at `i=0`: pick any vector not parallel to `tangent_0` as a reference (world Z, or world X if tangent is near-vertical — same "avoid near-parallel reference" pattern already used in the current `BarTriangulator._cylinder_segment`), then Gram-Schmidt it against `tangent_0` to get `normal_0`; `binormal_0 = cross(tangent_0, normal_0)`.
2. Propagate frame `i → i+1` by rotating `normal_i`/`binormal_i` by the minimal rotation that takes `tangent_i` to `tangent_{i+1}` (Rodrigues' rotation formula around `axis = normalize(cross(tangent_i, tangent_{i+1}))`, angle `= acos(clamp(dot(tangent_i, tangent_{i+1}), -1, 1))`). When the two tangents are parallel (straight run), the rotation is the identity — frame carries straight through unchanged, exactly the "no twist on straight segments" property Frenet lacks.
3. No end-to-end twist correction (a closed-loop stirrup could accumulate a small holonomy mismatch between the seed and final frame): acceptable for this project's shapes (predominantly open paths); if closed-loop artifacts appear in practice, evenly redistributing the mismatch angle across all rings is the standard fix, deferred until it's an observed problem rather than a hypothetical one.

### Stage 4 — Rings
For each sample `i`, generate `N_SIDES` vertices (matching the existing `segments=12` default, kept as a parameter) around the circle defined by `center=P_i`, radius `bar.radius`, using `normal_i`/`binormal_i` as the two in-plane basis vectors — same trigonometric construction the current `_cylinder_segment` already uses per-segment, just driven by the propagated frame instead of a locally-recomputed one.

### Stage 5 — Bridge rings
For each consecutive ring pair `(ring_i, ring_{i+1})`, connect vertex `j` to `j+1` (mod `N_SIDES`) across both rings with 2 triangles per side face — identical topology to the current per-segment side-wall triangulation, just applied ring-to-ring across the whole path instead of per independent segment.

### Stage 6 — End caps
Exactly 2 fan-triangulated caps total per bar (one at `ring_0`, one at `ring_{N-1}`), not one pair per segment. This is the actual fix — everything else in stages 4/5 is topologically similar to what exists today, but today's version repeats caps at every interior joint; the continuous version has interior joints with no caps at all, just a shared ring bridged on both sides.

**Closed paths** (`BarPath.closed`, e.g. stirrup): no caps at all — bridge the last ring back to the first ring instead, forming a torus-like closed tube.

### Stage 7 — Normals
Per-vertex normals averaged from adjacent face normals (standard smooth-shading normal generation), rather than each ring vertex only knowing about its own segment's local frame. Needed for correct rendering (Phase 10G viewer) and is essentially free once faces are known.

## Manifold guarantees (acceptance criteria, testable)
For a reconstructed bar with N points (N-1 segments, open path):
```
vertices = N * N_SIDES + 2                (rings + 2 cap centers)
faces    = (N-1) * N_SIDES * 2            (side walls)
         + (N_SIDES - 2) * 2              (2 fan-triangulated caps... see note)
```
Note: a fan cap from a center vertex to an N_SIDES-gon is exactly `N_SIDES` triangles (not `N_SIDES-2`) when using a center vertex, matching the current cap approach (`_cylinder_segment` already does this: `start_center`/`end_center` fan). So: `faces = (N-1)*N_SIDES*2 + N_SIDES*2` (2 caps × N_SIDES triangles each).

Testable, mesh-level acceptance criteria (extending the existing `_mesh_edge_report` open/nonmanifold-edge check, which will now be a *meaningful* pass/fail instead of trivially passing on independently-valid segments):
- **0 open edges, 0 nonmanifold edges** (existing check, but now actually exercising cross-segment topology).
- **Exactly 1 connected mesh component per bar** (new check — the old approach could in principle produce N-1 touching-but-topologically-separate solids that happen to share vertices without sharing edges; the continuous version genuinely shares edges between rings, so a flood-fill over the face-adjacency graph should find exactly one component).
- **0 duplicate vertex positions** beyond the intentional ring/cap sharing (i.e., no two rings emitted at the same location unless the input path genuinely revisits a point).
- **Exactly 2 cap fans for an open path, 0 for a closed path** (new check — directly tests that Stage 6 was actually applied once per bar, not once per segment).
- **Vertex/face counts match the formula above exactly** for every bar (regression-style check, catches any topology drift immediately).

## `BarPath` provenance additions (per your request)
Two fields, both independent of engineering confidence (which lives on `EngineeringFamily`/`PhysicalBar` and reflects annotation/association/spacing trust — a completely different question from "how faithfully was this geometry recovered"):
```python
recovery_confidence: float   # 1.0 = simple_path/closed_loop, reduced for longest_path_in_branch
                              # proportional to (edges_used / total_edges), 0.5 fallback for fallback_straight
recovery_notes: List[str]    # human-readable, e.g. "1 edge excluded from branch (degree-3 node at ...)"
```
`recovery_confidence` is a property of Stage 10B (geometry recovery), not this sweep stage — implemented alongside these mesh changes since both were requested together, but conceptually separable.

2. `BarTriangulator` (or a new class replacing it, keeping the old one for a version comparison during rollout) calls `tube_sweep` instead of per-segment `_cylinder_segment`.

4. Add `recovery_confidence`/`recovery_notes` to `BarPath`, populated in `geometry_recovery.py`.
5. Re-run the full audit trail (10.0-style checks) against the new mesh output; update `docs/audits/phase10/10.0_reconstruction_audit.md`'s 10E status.
