# Phase 10.3 — Continuous Tube Sweep: Implementation & Verification

Implements the design in `10.2_continuous_tube_sweep.md` exactly as specified: parallel transport frames (not Frenet), one continuous mesh per bar, caps only at the two true endpoints of an open path (none for closed loops), plus `BarPath.recovery_confidence`/`recovery_notes` as a dimension independent of engineering confidence.

## What changed
- **New `core/reconstruction/tube_sweep.py`**: `sweep_bar_path(points, radius, closed, segments)` implements Stages 2-7 from the design doc — tangents (central/forward/backward difference), parallel transport frame propagation via Rodrigues' rotation (minimal rotation frame-to-frame, identity on straight runs), rings, ring-bridging side walls, exactly 2 end caps for an open path / a closing bridge for a loop, no per-segment capping.
- **`BarTriangulator` rewritten** to call `sweep_bar_path` once per bar instead of `_cylinder_segment` once per centerline segment.
- **`BarPath` gained `recovery_confidence`/`recovery_notes`**, populated in `geometry_recovery.py`: 1.0 for a fully-recovered simple path or closed loop, `edges_used/total_edges` for a truncated branch, 0.5 for the straight-line fallback. Propagated correctly through `ReinforcementAdjuster` (fixed the same "new field dropped on rebuild" mistake caught once already in 10.1, this time before it shipped).
- **`run_phase10.py` mesh audit extended**: added `_mesh_topology_report()` (union-find connected-component count per mesh) and a new validation line, "One continuous mesh per bar," alongside the existing open/nonmanifold-edge check.

## Verified results

**Numeric, exact**: vertex/face counts dropped from 2912/5376 (10.1's per-segment-capped output) to **1428/2832** for the identical underlying geometry — the difference is exactly the redundant internal caps at the 55 interior joints (57-segment N6 path + 19-segment N7 + 4×9-segment N4) that no longer exist. Verified against the design doc's closed-form formula for all 6 bars independently (e.g. N6: 58 points × 12 sides + 2 = 698 vertices, 57×24 + 24 = 1392 faces) — matches exactly, both individually and summed.

**Topology, not just counts**: `Multi-Component Meshes: 0` (new check) confirms every bar is genuinely one connected mesh, not several solids touching at shared vertices. `Watertight: True` held both before and after — the old check was too weak to distinguish continuous-vs-segmented, which is exactly why the new connected-component check was worth adding.

**Visual proof** (`debug/phase10/n6_tube_render.png`, rendered directly from mesh faces, not the centerline): N6's 58-point branch-derived bar renders as one smooth, continuously-bending tube around its hook, with no visible seam, step, or doubled surface at the bend — the qualitative target the design doc set out.

## Deferred, per the design doc
- Closed-loop (stirrup) frame continuity (potential small holonomy mismatch at the seed/final frame junction) — no stirrup family exists in the current dataset to test against; flagged as a future fix triggered by an observed artifact, not a hypothetical one.
- Curvature-adaptive resampling of Stage 1 — not needed to fix the manifold-joint problem (the actual defect), so deliberately out of scope here.
