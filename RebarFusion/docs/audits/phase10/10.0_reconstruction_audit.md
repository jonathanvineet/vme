# Phase 10.0 — Reconstruction Audit (Freeze Candidate)

Scope: `core/reconstruction/{assembly_builder,bar_builder,mesh_builder,triangulator,adjustments,models}.py`, `run_phase10.py`, run against the 3 families currently produced by the frozen Phase 9 pipeline. Purpose: measure what fraction of the reconstruction pipeline is already correct before adding any new features, per your instruction. **This audit found real, fixed bugs (see the 3 crash/gate fixes from the previous round) and, in this pass, one major unfixed fidelity gap plus two secondary trust issues — Phase 10 is not ready to freeze.**

## 10A — Coordinate Frames: **PASS**
`AssemblyBuilder._coordinate_frame` builds `u = (cos θ, sin θ, 0)`, `v = (-sin θ, cos θ, 0)`, `w = (0,0,1)` from the family's dominant direction — a closed-form trig construction, not empirically fitted, so orthogonality and right-handedness are guaranteed by construction rather than approximately true. Verified numerically anyway on both assemblies in the current dataset:
```
|u·v| = 0.0e+00   |u·w| = 0.0e+00   |v·w| = 0.0e+00
|u| = |v| = |w| = 1.0000
handedness error (|u×v − w|) = 0.0e+00
```
Exactly zero error, both assemblies, matching the closed-form expectation. Deterministic (same UUID/derivation scheme as the rest of the pipeline). No issues found.

## 10B/10C — Centerlines & Physical Bars: **FAIL — major fidelity gap**
`PhysicalBarBuilder._bar_from_member` builds **every** bar's centerline the same way, regardless of shape:
```python
raw_points = [(cx - ax*half, cy - ay*half, 0.0), (cx + ax*half, cy + ay*half, 0.0)]
```
a straight 2-point line through the member centroid, using only `family.orientation` and `family.length`. It never reads the member's actual per-edge geometry. Confirmed against real data — every reconstructed bar in the current dataset, regardless of recognized shape, comes out as a straight single-segment line:
```
mark=N6  family_shape=branch  n_points=2  n_segments=1
mark=N7  family_shape=branch  n_points=2  n_segments=1
mark=N4  family_shape=l_bar   n_points=2  n_segments=1   (×4)
```
None of these families are `straight_bar` type. `l_bar` should have a bend; `branch` should have a junction. Both get flattened into a straight line matching only total length — the actual shape recognized by Phase 7 (with real geometric evidence) is silently discarded at this stage. **This is not a hypothetical concern** — with the current dataset, 100% of reconstructed bars are shape-flattened, because 0 of the 3 families are `straight_bar` type.

`BarPath.closed` is set correctly (`recognition_type == "stirrup"`), but since points are always exactly 2, closing a stirrup would currently produce a degenerate 2-point closed loop (a segment closed back on itself) rather than a real rectangular/hook tie shape — untested with current data (no stirrup family exists yet) but a predictable consequence of the same root cause.

## 10D — Assembly Conservation: **PASS**
Traced Family → Members → Bars → Meshes for all 3 families: 6 members detected, 6 bars built, 6 meshes generated, `Families With Matching Bar Counts: 3/3`. Nothing disappeared. This check was already implemented in `run_phase10.py`'s own validation and confirmed correct.

## 10E — Mesh Validation: **PASS** (for the geometry currently produced)
Checked every mesh for degenerate triangles (repeated vertex index), zero-area faces (cross-product magnude ~0), and duplicate vertex coordinates, beyond the existing open/nonmanifold-edge check:
```
6/6 meshes: 26 vertices, 48 faces, 0 degenerate faces, 0 zero-area faces, 0 duplicate vertices
```
Clean. **Caveat**: `BarTriangulator.triangulate()` caps every centerline *segment* independently (`start_center`/`end_center` per segment, not shared across segments) — for a genuinely multi-segment centerline (which 10B/10C currently never produces), consecutive segments would each get their own end caps at the joint, meeting tip-to-tip rather than forming one continuous swept shape. This would likely still pass the open/nonmanifold-edge check (each segment is independently a valid closed cylinder), so it's a **latent, currently-unexercised bug** that will surface the moment 10B/10C is fixed to emit real multi-segment/bent centerlines. Flagging now so it isn't rediscovered later as a "new" bug.

## 10F — Reconstruction Fidelity: **FAIL — two trust issues, not geometry bugs**
Compared family → bar for length, diameter, and confidence:
```
mark=N6  length_err=0.0000mm  diameter_err=12.0000mm  confidence: bar=0.000 family=0.000
mark=N7  length_err=0.0000mm  diameter_err=12.0000mm  confidence: bar=0.000 family=0.000
mark=N4  length_err=0.0000mm  diameter_err=0.0000mm   confidence: bar=0.637 family=0.637 (×4)
```
- **Length "preserved" is tautological, not an independent check**: `bar_builder` derives the bar's length directly from `family.length` by construction (see 10B/10C), so this comparison can never fail regardless of whether the reconstruction is actually correct. Not a real fidelity signal as currently wired.
- **Diameter fabrication, silent**: `diameter = float(getattr(family, "diameter", 0.0) or 12.0)` — when a family has no diameter data (N6, N7 both do, since branch-type families rarely get a `TOKEN_DIAMETER` association), the bar silently gets a **fabricated 12mm default**, with nothing in the output distinguishing a real annotated diameter from a made-up one. A viewer or export consuming `bars.json`/the OBJ mesh has no way to tell these apart.
- **Zero-confidence bars render identically to real ones**: N6 and N7 both have `confidence: 0.000` (consistent with Phase 9's single-member-family finding), yet Phase 10 still generates full, clean, watertight 26-vertex meshes for them with no visual or data flag. A viewer showing these meshes next to N4's (confidence 0.637) would present all three with equal visual authority.

## 10G — Determinism: **PASS, now formalized**
Added `tests/determinism_phase10.py` (mirrors `tests/determinism.py`'s pattern) — runs the full pipeline through Phase 10 mesh generation N times and hashes `assemblies.json` + `bars.json` + `meshes.json` + the first OBJ file's bytes.
```
5/5 runs: identical hash
```
Confirmed via `sha256sum` on the OBJ output directly, and via a proper `hashlib`-based JSON hash (an initial check using Python's built-in `hash()` on JSON strings gave a false nondeterminism signal — that's due to `hash()`'s own per-process string randomization, not the pipeline; corrected before concluding). No `uuid4()` calls exist anywhere in `core/reconstruction/` (checked directly), consistent with the clean result.
