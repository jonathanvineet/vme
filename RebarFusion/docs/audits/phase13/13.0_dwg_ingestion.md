# Phase 13.0 — DWG Ingestion

**Scope: unlock the 9 `.dwg` files in `test_project/` — unreadable for the entire life of this project, ranked as the #1 risk in the Phase 12 research report — without creating a second geometry parser.** The DWG path is: ODA File Converter → temporary DXF → the existing, frozen `DXFReader` → the existing, frozen canonicalizer. One extraction path for both formats.

## The converter

- **ODA File Converter 27.1.0.0** (Open Design Alliance's free distribution), downloaded from `opendesign.com/guestfiles/oda_file_converter` (macOS arm64 `.pkg`), **extracted locally into `tools/oda/ODAFileConverter.app` via `pkgutil --expand` — not installed system-wide**. The exact converter version travels with the repo checkout; `RF_ODA_CONVERTER` overrides the binary path for other environments.
- Invoked headless in batch mode (`<in_dir> <out_dir> ACAD2018 DXF 0 1 *.dwg`); output is DXF R2018 (AC1032), which ezdxf reads natively.
- **Raw-output determinism, measured before any code was written**: converting all 9 DWGs twice produced DXF pairs that differ in exactly 8 lines each — the `$TDUPDATE`/`$TDUUPDATE` header variables, which ODA stamps with conversion time. All entity content is byte-identical, and the canonicalizer never reads those headers. Canonical-level determinism is therefore exact, and is regression-tested (below) by clearing the cache and forcing a genuine reconversion, not just re-reading a cached file.

## What was built

- `core/readers/dwg_converter.py` — `convert_dwg_to_dxf(path)`: stages the single DWG in a temp dir (ODA's CLI is batch-only), converts, and caches the result keyed by the DWG's SHA-256 in the user cache dir (`~/.cache/rebarfusion/dwg2dxf/`). The cache deliberately lives **outside** any project directory: Phase 1's `load_directory()` walks the project tree recursively and would scan a cache subfolder's DXFs as new drawings.
- `core/readers/dwg_reader.py` — `DWGReader`: every `read_*` method converts (cached) and delegates to `DXFReader`. `can_read()` returns `False` when the converter binary is absent, so converter-less environments degrade to Phase 1's existing "No reader available" warning instead of erroring mid-read.
- `core/project.py` — one additive change to the Phase 1 reader registry: `[DXFReader()]` → `[DXFReader(), DWGReader()]`.
- `core/full_pipeline.py` — reader dispatch per drawing (was a hardcoded `DXFReader`), plus resolving the converted-DXF path once and handing it to `canonicalize()`, which independently re-opens the file with ezdxf and would otherwise be handed a raw `.dwg`. These are the only two frozen-file edits, both dispatch plumbing, neither touching parsing/canonicalization logic.
- `tests/test_dwg_ingestion.py` — 4 checks (converter present, reader dispatch, reconversion determinism at canonical level, real-data regression on `SS-GF-01(R).dwg`). All pass.
- `tests/fixtures/single_drawing/` — a copy of `SS-GF-01(M).dxf`. The frozen Phase 12.1-12.4 test suites assert exact single-readable-drawing results; now that `test_project` has 8 readable drawings, those suites run against this fixture (all 4 still pass unchanged there). The frozen assertions were preserved, not rewritten to chase the new data.

## What the unlocked data showed — findings, ranked

1. **The research report's central prediction is confirmed with real data.** `SS-GF-01(R).dwg` — whose unreadability forced the report to reason by analogy — contains **exclusively self-decoding T-marks** (18 families: T8/T10/T12/T20, with real diameters and spacings), and **zero N-reference-codes**. The Mould-vs-Reinforcement sheet convention, inferred from PW-GF-09's PDFs and `SS-GF-01(M)`'s vocabulary, is exactly right.
2. **The first real `PhysicalIdentity` exists.** Across 8 drawings / 88 observations / 1100 deduplicated pair-decisions, exactly one pair reached ACCEPTED: two `T16` families, both from `PW-GF-09(R).dwg`, unanimous conflict-free evidence (same mark, same diameter/spacing/shape/orientation). **Open QA item, flagged rather than celebrated**: both observations are from the *same sheet*, so this is either the same physical bars drawn in elevation + section (legitimate same-sheet cross-view fusion — the `(R)` sheet genuinely contains both view types) or two distinct T16 bar groups sharing a mark (a false merge). Needs verification against the drawing before this identity is trusted; recorded in `known_limitations.md`.
3. **The cross-drawing case behaves correctly.** All `SS-GF-01(M)`↔`SS-GF-01(R)` pairs land in REVIEW — the system sees shape/orientation agreement between `N6`/`N7` (mould) and various T-mark families (reinforcement) but a mark conflict, and correctly refuses to auto-merge. Resolving N-code → T-mark identity requires the reference-code lookup mechanism (schedule/annotation evidence categories, named but generator-less in Phase 12.3) — the architecture predicted precisely this gap, and the REVIEW outcomes are it saying "plausible, not provable with current evidence," which is the designed behavior, not a failure.
4. **Duplicate handling worked, with one cosmetic quirk**: `banana.dwg`, `PW-GF-02(M1).dwg`, and `PW-GF-02(M1)-copy.dwg` are byte-identical; Phase 1 kept whichever `os.walk` yielded first (`banana.dwg`) and marked the properly-named originals as duplicates. Pre-existing Phase 1 behavior, not introduced here — but it means 3 observations carry `banana.dwg` as their source (and an `unclassified` role, since the filename defeats the identity parser). Worth a Phase 1 tie-break rule (prefer the name that parses) eventually; cosmetic today.

## What could not be done

- **Original Step 5 (same drawing in both DWG and DXF, compared through the pipeline)**: impossible with this dataset — no drawing exists natively in both formats (`SS-GF-01(M)` is DXF-only; every DWG has no DXF sibling). The closest available evidence is the reconversion-determinism check plus the fact that converted DXF flows through the identical parser/canonicalizer as native DXF. Honestly noted rather than substituted with a weaker check dressed up as the real one.
- The 4 PDFs remain unread by the pipeline (they are print versions of the same PW-GF-09 sheets now readable as DWG, so nothing unique is lost).

## Freeze criteria

- [x] Converter locally extracted, version-pinned (27.1.0.0), no system install, path overridable.
- [x] One geometry parser: DWG path delegates to frozen `DXFReader`/canonicalizer; AST of new modules imports no parsing logic of its own.
- [x] Reconversion determinism at canonical level (cache cleared between runs, hash identical).
- [x] Real-data regression on `SS-GF-01(R).dwg` (18 observations, T-marks only, `reinforcement_typical`).
- [x] Frozen Phase 12.1-12.4 suites still pass against `tests/fixtures/single_drawing/`.
- [x] Full Phase 1→12 pipeline runs end-to-end on the real 8-drawing package (≈37s).

**Phase 13.0 is frozen.** The project's oldest structural limitation — single-drawing validation — is lifted: every correctness claim can now be re-examined against 8 real drawings, and Phase 12 has real multi-view data to reason over.
