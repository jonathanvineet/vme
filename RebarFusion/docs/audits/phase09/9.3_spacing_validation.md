# Phase 9.3 — Spacing Validation Audit

Scope: `core/engineering/spacing.py` (new), wired into `run_phase9.py`, run against `SS-GF-01(M).dxf`'s 3 current families.

## Methodology check first — measurement was already correct
Before building anything, checked whether spacing used naive centroid distance (the failure mode you flagged) or true normal-projected distance. Traced `FamilyBuilder._discover_members`/`_profile` (`core/engineering/family.py`): `offset = profile.perp_center - rep_profile.perp_center`, where `perp_center = cx*px + cy*py` and `(px, py)` is the perpendicular unit vector to the bar's own orientation. **This is already a signed projection onto the family's normal direction, not Euclidean centroid distance.** No fix needed here — reporting this honestly rather than re-implementing something that already worked, and redirected effort to what was genuinely missing: per-gap provenance and real statistics.

## What was actually missing
`run_phase9.py` only reported one pooled global average ("Average Spacing Error: 60.436mm" as a single number across all families). No per-gap record existed, no RMSE/bias/std-dev, no outlier isolation.

Built `core/engineering/spacing.py`:
- `SpacingMeasurement` — one record per consecutive member pair: both member UUIDs, both offsets, measured gap, reference spacing, signed residual, residual %, `is_outlier`, measurement method.
- `FamilySpacingStatistics` — per family: gap count, mean/median spacing, **RMSE**, **bias** (mean signed residual — reveals systematic over/under-measurement vs. pure noise), **std dev**, max/min absolute error, outlier count.
- Outlier threshold reuses the existing 35%-of-spacing tolerance already used by `FamilyBuilder._select_spacing_sequence` to decide sequence membership, rather than inventing a second number.

New outputs: `spacing_measurements.json`, `spacing_statistics.json`, `spacing_outliers.json`, `spacing_histogram.json` (50mm-bucketed), plus a per-family gallery (`debug/phase09/<drawing>/spacing/<mark>_<uuid8>.png`) drawing each gap with its measured distance, red for outliers.

## Result on this drawing
Only 1 of 3 families has ≥2 members (spacing requires at least a pair): `N4`, 4 members, 3 gaps.

```
gap 1: 295.8mm  (not outlier)
gap 2: 477.1mm  (OUTLIER — residual +181.3mm, +61.3% vs. reference)
gap 3: 295.8mm  (not outlier)
RMSE: 104.7mm   bias: +60.4mm   std dev: 85.5mm
```

**The single outlier is fully explained, not a mystery**: two gaps are exactly 295.8mm and the third is 477.1mm — not a clean multiple of the regular spacing, meaning a member is missing between those two positions. This corroborates, independently, what `FamilyQA` had already flagged for this same family via a completely different code path (`missing_member_offsets`/`spacing_gaps_detected`). Per your own acceptance criteria checklist — wrong family? no. wrong annotation? no. wrong member? no. **missing bar? yes, and now precisely quantified** (181mm, 61%) instead of just a boolean flag. Geometry/inference bug? No evidence of one.

## Important caveat — don't over-read this result
This is the *only* family in the drawing with measurable spacing, and it happens to be built entirely from the same **5.34mm-length artifact cluster** already flagged as "review" by Phase 7.6 and independently corroborated by Phase 9.2 (both audits, this family, same components). That means:
- The spacing **measurement methodology** is validated: it correctly computed gaps, correctly identified the one true outlier, and correctly explained it as a missing member.
- It does **not** validate real-world spacing **accuracy** against genuine, uncontested reinforcement, because there is no other example in this drawing to check against yet. If the underlying 5.34mm cluster turns out to be a real artifact (not real bars) once that open question is resolved, this entire spacing example evaporates along with it.
- This isn't a new problem — it's the same open item from Phase 7.6/9.2, now touching a fourth phase's output. Worth noting for whoever eventually resolves the artifact question: fixing it will also remove Phase 9's only current spacing data point, and a genuinely clean drawing (or a second test drawing) would be needed to properly validate spacing accuracy going forward.

## Verdict
Spacing measurement methodology confirmed correct (normal-projected, not Euclidean) and now has full per-gap provenance, real statistics (RMSE/bias/std-dev, not a single pooled average), and visual outlier galleries. The one outlier found is fully explained as a missing member, cross-validated against an independent QA path. No spacing-inference bug found — but the dataset available in this drawing is too thin (1 family, 3 gaps) and too entangled with the still-open 5.34mm artifact question to declare spacing broadly validated. Recommend treating this stage as "methodology proven, awaiting a clean or larger dataset" rather than "fully validated," and resolving the 5.34mm artifact question before leaning on spacing numbers for anything downstream.
