# Phase 9.2 — Standalone Object Provenance Audit

Scope: `_classify_standalone` and `_write_standalone_inspector` (`run_phase9.py`), run against `SS-GF-01(M).dxf`'s current 34 standalone engineering objects (post Phase 7.6 plausibility filtering, post Phase 9.1 mark-broadcast fix).

## What changed
`_classify_standalone` already computed a rich comparison against the nearest family (distance, orientation delta, length delta, layer/type match) — but only surfaced it as a formatted text string (`"orientation differs 72.6 degrees from N4"`), discarding the underlying numbers. Refactored to return structured `evidence` alongside the reason/details, and:
- Added `debug/phase09/<drawing>/standalone_provenance.json` — one entry per standalone object with `nearest_family_uuid`, `nearest_family_mark`, `distance_mm`, `family_layer`/`family_recognition_type`/`family_length`, `orientation_delta_deg`, `length_delta_pct`, plus the object's own recognition type/layer/length/confidence.
- Reorganized the per-object gallery cards from a flat `standalone/NNNN.png` into `standalone/{family_rejected,orientation_rejected,length_rejected,isolated}/NNNN.png`, matching the requested bucket structure.

## Finding 1 — "Different orientation" (15/15) is the same artifact, independently confirmed
Cross-checked against `standalone_provenance.json`: **all 15** "Different orientation" objects have `length: 5.34` and `length_delta_pct: 0.0` against family N4 — i.e., they are length-identical to the N4 representative but sit at `orientation_delta_deg: 72.6` from it. This is the exact same 5.34mm artifact type flagged by the Phase 7.6 plausibility audit (which put it in "review," not "reject," because its ratio-to-median score narrowly missed the reject cutoff). Two independent phases — Phase 7.6's length-implausibility check and Phase 9.2's family-comparison check — now agree on the same 15 (of the reported 19 "review" instances; 4 apparently became family members or were otherwise absorbed rather than standalone) suspect components, via completely different methods. That agreement is itself meaningful evidence this cluster is a real artifact, not two false alarms. Still not deleted or force-rejected — flagged, visible, and now doubly corroborated for whoever makes the domain call.

## Finding 2 — "Different type" (12/12) is legitimate, not a bug
Sampled several: e.g. a `straight_bar` 152mm from the nearest family, which happens to be `branch`-type `N6`. This is not "should have joined N6 but got excluded by a bug" — it's correctly reporting that the geometrically-nearest family is simply a different shape, so there's no compatible family for this component regardless of distance. No fix needed here; the classification is doing its job.

## Finding 3 — Branch (5) and Stirrup (2) are pass-through classifications, not compared
`_classify_standalone` returns early for `branch`/`stirrup` recognition types without ever comparing to a family — these types are always standalone by design in the current builder (no branch/stirrup family concept exists yet). Not a defect, just worth naming explicitly: the "why standalone" answer for these 7 objects is simply "this recognition type is never grouped into a family," not a rejected comparison.

## Verification

- `tests/regression.py test_project`: Phase 2/3/6/7 all pass; only the pre-existing, deliberately-open Phase 6 duplicate-edge item remains.

## Verdict
The standalone bucket is now fully explained with numeric provenance, not just category labels — every one of the 34 objects has a precise, inspectable reason. Nothing here indicates a new bug: "Different orientation" corroborates (doesn't newly reveal) the Phase 7.6 artifact cluster, "Different type" is working as intended, and "Branch"/"Stirrup" are an explicit design gap (no family concept for those shapes yet) rather than a mistake. No source-level fix is needed from this audit alone — the actionable item remains the same one Phase 7.6 already flagged (the 5.34mm `l_bar` cluster), now with stronger, cross-phase evidence behind it.
