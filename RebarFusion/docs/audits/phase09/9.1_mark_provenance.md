# Phase 9.1 — Mark Provenance Audit

Dedicated audit only, per instruction — no source changes in this round. Traces exactly where every `mark="N6"` (and, in the process, `mark="N4"`) assignment comes from, before touching any propagation/family logic.

## Correcting the working hypothesis
The original suspicion was that `FamilyBuilder._seed_bars()` was propagating marks too aggressively. Tracing it precisely shows **that hypothesis was wrong** — `_seed_bars()` never assigns `.mark` to anything. Its `seeds` dict comprehension only *includes* bars that already have a truthy `.mark`, and its two propagation loops only fill in missing `.diameter`/`.spacing` on bars that are already in that mark-keyed dict; the `mark` field itself is never written there. So whatever is producing 3 separate `N6`-marked components has to originate upstream, in Phase 8.

## The real mechanism
`MarkConstraint.apply()` (`core/engineering/models.py`) sets `obj.mark = self.token.value` directly — and a `MarkConstraint` is created for **every** `AssociationCandidate` with `score >= 0.5` for a `TOKEN_MARK` token (`EngineeringAssociationEngine.build_constraints`). Critically, `find_group_candidates` returns up to `k=5` candidate *components* for a single annotation group/token — it was designed for tokens like length/diameter that can legitimately apply to several nearby bars, but `build_constraints` applies that same fan-out logic to `TOKEN_MARK` too, with no restriction to "the one component this mark actually identifies."

Traced with full provenance for all 3 `N6` components — **all three originate from the exact same single annotation group** (`group#19`, text content `['N6', '', '', '']`):

| Component | Score | Recognition type | Length (mm) |
|---|---:|---|---:|
| `5c1cda77...` | 0.950 | `branch` | 3227.8 |
| `f3e639ef...` | 0.950 | `branch` | 1964.8 |
| `0d40eb22...` | 0.536 | `straight_bar` | 56.2 |

One `N6` text annotation → three independent `MarkConstraint`s → three unrelated components all carrying `mark="N6"`. Two are `branch`-type components differing in length by 1263mm; the third is a totally different shape type and orientation. These are not plausibly the same physical bar family.

## A second, distinct anomaly found while tracing N4
The `N4` broadcast's three target components report `profile.length` of **5.3mm, 5.3mm, and 0.5mm** — physically nonsensical for a structural bar (for comparison, the `N6` targets are 1964-3227mm, which is a plausible rebar length range). This means `_component_profile`'s length computation is producing garbage for at least these components, independent of the mark-broadcast bug. Not yet traced further — flagging as a second, separate defect that needs its own investigation (likely in how `_profile`/`_component_profile` computes `length` for `l_bar`/certain `straight_bar` components), since a correct fix to mark broadcasting won't fix this.

## Not a chain — a fan-out
Your hypothesized failure mode was a recursive propagation chain (`A → B → C → D`). That's not what's happening here: it's a single annotation fanning out to `k` candidates in one step via `find_group_candidates`/`build_constraints`, not a multi-hop chain through `_seed_bars`. Worth being precise about, since the fix is different: this needs to change how `TOKEN_MARK` candidates are converted into constraints (e.g. apply a mark to only the single best-scoring candidate, or require the candidates to pass a shape/type-similarity gate before more than one can share a mark), not anything in `_seed_bars` or `FamilyBuilder` at all.

## Recommendation
Root cause is now precisely located: `EngineeringAssociationEngine.build_constraints` treats `TOKEN_MARK` the same as `TOKEN_LENGTH`/`TOKEN_DIAMETER` (fan out to every candidate ≥0.5), when a mark should almost certainly identify at most a small, shape-consistent set of components, not any of the top-5 nearest by distance. Suggested fix shape (not yet implemented, awaiting sign-off): either (a) restrict `TOKEN_MARK` constraints to the single highest-scoring candidate per group, or (b) keep multi-target mark assignment but require candidates to pass a shape/type/length-similarity gate against each other before more than one can share a mark from the same annotation. Separately, the N4 length anomaly (5.3mm/0.5mm) needs its own trace into `_component_profile` before Phase 9 metrics involving length can be trusted at all.
