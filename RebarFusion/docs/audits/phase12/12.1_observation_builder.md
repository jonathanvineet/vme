# Phase 12.1 — Observation Builder

**Scope, per `docs/research/phase12_cross_view_fusion_research.md` (Addendum 1/2) and the roadmap that followed it: build the engine that turns each drawing's Phase 9 output into `PhysicalObservation` records. Nothing else.**

Implementation Rule #1 (as specified before this subphase started): the system is forbidden from generating fusion candidates, scoring evidence, resolving identity, or reconstructing geometry in this subphase. The only output is a flat list of `PhysicalObservation`. That rule is honored — `core/fusion/models.py` defines only `DrawingRole` and `PhysicalObservation`; `ObservationEdge`, `Claim`, and `PhysicalIdentity` (Phase 12.2-12.4) are not implemented yet, not even as empty stubs.

## What was built

- `core/fusion/models.py` — `DrawingRole` (+ `classify_drawing_role`) and `PhysicalObservation` (+ `classify_mark_namespace`).
- `core/fusion/observation_builder.py` — `build_observations(results)`: one `PhysicalObservation` per `EngineeringFamily`, across however many `PipelineResult`s (one per readable drawing) are passed in. Deterministic UUIDs (`NAMESPACE_OBSERVATION`, keyed on `filename|family.uuid`) and deterministic output ordering, matching every other phase's determinism discipline.
- `run_phase12.py` — CLI entry, writes `debug/phase12/<project>/observations.json`.
- `tests/test_phase12_observation_builder.py` — 4 checks (determinism, real-data regression, synthetic multi-drawing shape, a guardrail against Phase 12.2+ concepts leaking in early). All pass.

Two small, narrow classifications were added, both directly evidence-grounded rather than invented:

- **`classify_drawing_role`** maps `DrawingIdentity.view` to `'reinforcement_typical'` (`R`) / `'mould_instance'` (`M`, `M1`, `M2`, ...) / `'unclassified'`, based on the convention confirmed by direct inspection of `PW-GF-09(R)/(M1)/(M2)` in the research report. This is new code in `core/fusion/`, not a change to `core/identity_parser.py` — the research report flagged `view` as semantically opaque everywhere in the codebase; this is the first (and only, so far) place that opacity is resolved, and it's additive, not a modification of frozen Phase 1 code.
- **`classify_mark_namespace`** reuses `AnnotationParser`'s existing `DIA_PATTERN`/`MARK_PATTERN` regexes (not reinvented) to tell self-decoding marks (`T12`) from reference-code marks (`N7`) — the distinction the research report's Step 1/Step 4 identified as necessary and the original data model got right on paper; this is that classification actually running against real family output for the first time.

## Real-data result

Only one drawing in `test_project/` is machine-readable: `SS-GF-01(M).dxf`. Running `run_phase12.py test_project` against it produces exactly 3 observations — `N4`, `N6`, `N7` — every one correctly classified `drawing_role.role == 'mould_instance'` and `mark_namespace == 'reference_code'`. This is the Observation Builder independently reproducing, from real pipeline output rather than manual inspection, the same conclusion the research report reached by reading the drawing's text by hand: this drawing is a Mould drawing, and its marks are reference codes, not self-decoding bar marks.

One honestly-reported surprise, not investigated further here because it's out of this subphase's scope: `N4`'s observation carries `diameter=8.0`, `diameter_source='family_annotation'` — i.e. `EngineeringFamily.diameter` is non-null for `N4`, whereas `docs/known_limitations.md` documents `N4` as entangled with the Phase 7.7 fragment cluster and `docs/audits/phase10/10.1_geometry_recovery_redesign.md` records `N4`/`N6`/`N7` as generally lacking real diameter data. This isn't a new bug introduced here — the Observation Builder only reads `family.diameter` as-is — but it is worth flagging forward: whatever produced `N4`'s `diameter=8.0` needs to be traced (likely a `DIA_PATTERN` match against a nearby, unrelated `T8`-shaped annotation the fragment cluster is entangled with) before Phase 12.3 (Evidence Engine) could safely treat it as a diameter claim. Recorded here rather than silently accepted.

## The real-data constraint, and how it was worked around

`test_project/` currently has **zero elements with two machine-readable drawings** — `SS-GF-01`'s `(R)` companion is an unreadable DWG, and `PW-GF-09` (the set the research report read visually from PDF) has no DXF at all. That means a genuine end-to-end, real-file test of "does the Observation Builder correctly produce observations from two different drawings of the same element" isn't possible yet, and won't be until DWG support exists (research report Step 9, ranked as the top risk).

This subphase doesn't block on that. `check_synthetic_multi_drawing` in the test file hand-builds two in-memory `PipelineResult`s (a `T12`-marked family on a `view='R'` drawing, an `N7`-marked family on a `view='M'` drawing) to prove the builder correctly differentiates `drawing_role`/`mark_namespace` across drawings before real fixture data exists — explicitly labeled synthetic in both the test docstring and this document, not presented as validated against real cross-drawing input.

## Revision — facts[] replaces flat aspect fields

Reopened before Phase 12.2 started, in response to review: the original `PhysicalObservation` (`mark`, `diameter`, `spacing`, `shape`, ... as separate optional fields) mixed two different questions — "what aspects can this observation possibly speak to" and "what value did it report" — into one flat, ever-growing field list. `facts: List[ObservationFact]` replaces all of it. Each `ObservationFact(aspect, value, confidence, source)` is both the capability declaration and the value in one: an aspect's presence in `facts` **is** the claim, so there's no separate `claims`/capability set that could drift out of sync with the actual values — deliberately, to avoid the exact duplicated-representation failure `core/full_pipeline.py`'s own docstring already warns about for drawing-level logic.

Consequences of the refactor:
- `diameter_source: 'family_annotation' | 'missing'` is gone. It's now implicit: a `DIAMETER` fact present means `family_annotation`; absent means the observation has no diameter claim at all. One less piece of state that could disagree with the data it was describing.
- Confidence is no longer a single per-observation number. Measured geometric aspects (`shape`/`orientation`/`position`/`length`/`quantity`) get `confidence=1.0` — read directly off recognized geometry, not inferred. Semantic aspects (`mark`/`diameter`) carry `family.confidence`, and `spacing` carries the more specific `family.spacing_confidence` — both genuinely uncertain, because they depend on Phase 8's annotation-to-geometry association being correct. This mirrors the recovery-confidence-vs-engineering-confidence split this project already established in Phase 9.4/Phase 10.1, applied per-fact instead of per-observation.
- `ASPECT_*` is an explicit, closed vocabulary (`mark`, `diameter`, `spacing`, `shape`, `orientation`, `position`, `length`, `quantity`). `HOOK` and `LEVEL`, named in the original research report's cross-view model, are intentionally **not** in this list — Phase 9's `EngineeringFamily` has no hook or absolute-level data to read yet, and adding them here would mean either fabricating a value or emitting a fact with `value=None`, both of which the observation invariant below forbids.

## Observation invariant (added this revision, applies retroactively and going forward)

**An observation must never carry a fact for an aspect it cannot directly read off its source.** Concretely: no `ObservationFact` is ever constructed with `value=None`; a family with no diameter produces zero `DIAMETER` facts, not a `DIAMETER` fact with an empty value. Verified by `check_observation_invariant` in the test suite three ways: (1) real data — `N6`/`N7` (whose families carry no diameter) must produce zero `DIAMETER` facts; (2) every emitted fact's `aspect` must be one of the declared `ASPECT_*` constants, so no ad-hoc aspect string can sneak in from a future edit; (3) no fact anywhere has `value=None`. This is also recorded as a standing acceptance criterion in the research report (`docs/research/phase12_cross_view_fusion_research.md`), so it constrains Phase 12.2-12.4 too, not just this subphase.
