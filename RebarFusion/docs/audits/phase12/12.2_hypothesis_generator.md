# Phase 12.2 — Hypothesis Generator

**Implementation objective, as specified before this subphase started: build a deterministic, exhaustive hypothesis generator that produces candidate identity groups from observations, but makes no decisions. No scoring. No acceptance. No merging. No geometry.**

## What was built

- `core/fusion/models.py` — added `HypothesisEvidence` (qualitative, no numeric score), `IdentityHypothesis` (`anchor_observation`, `candidate_observations`, `evidence`, `status`, always `PENDING` out of this subphase), and `HYPOTHESIS_PENDING/ACCEPTED/REJECTED` status constants (only `PENDING` is ever used by this subphase's code).
- `core/fusion/hypothesis_generator.py` — `generate_hypotheses(observations)`: one `IdentityHypothesis` per observation (as anchor), scoped to observations sharing the same `drawing_number` (Phase 1's existing grouping — a hard scope boundary, not a selectivity heuristic, since marks aren't globally unique across elements).
- `run_phase12.py` extended to also run hypothesis generation and write `debug/<project>/hypotheses.json`.
- `tests/test_phase12_hypothesis_generator.py` — 5 checks. All pass.

## The provenance field, added now per review

`ObservationFact` gained `uuid` and `source_entity_uuid` (Phase 12.1 was reopened a second time for this, alongside 12.2 — see `12.1_observation_builder.md`'s revision log). Populated honestly, not uniformly:

- **Populated** for measured/geometric facts (`shape`, `orientation`, `position`, `length`, `quantity`): `source_entity_uuid` = the family's representative component UUID — a real pointer into the CAD geometry.
- **`None`** for semantic facts (`mark`, `diameter`, `spacing`). Traced, not guessed: `core/engineering/solver.py::ConstraintSolver.solve()` holds the real originating entity (`EngineeringConstraint.token: AnnotationToken`, which has `source_uuid` pointing at the raw TEXT/MTEXT/DIMENSION entity) right up until `constraint.apply(obj)` — and then discards it. `EngineeringBar`/`EngineeringFamily` never receive it. This is a real, traced gap in already-frozen Phase 8 code, not something Phase 12 introduced. Wiring it through would mean modifying `core/engineering/solver.py` and `core/engineering/models.py::EngineeringConstraint.apply()`, which is out of scope for this subphase (the research report's constraint against modifying existing architecture without cause, and the explicit "no scoring/no decisions/no geometry" objective for 12.2 specifically). Flagged forward rather than worked around with a fabricated value.

## The candidate-generation ordering, implemented as specified

1. **Same `drawing_number`** — hard scope, computed once via `Dict[str, List[PhysicalObservation]]` grouping.
2. **`same_mark`** (namespace-aware — a `self_decoding` mark is never compared against a `reference_code` mark as the same kind of thing).
3. **`complementary_role`** (`mould_instance` <-> `reinforcement_typical`) — corroborating only, never sufficient alone.
4. **`fact_agreement`/`fact_conflict`** on any aspect both observations claim (excluding `mark`, handled by rule 2, and `position`, handled by rule 5) — agreement can qualify a pair on its own; conflict is always recorded, never used to exclude a pair from the evidence trail.
5. **`spatial_distance`** — always attached when both observations have a `POSITION` fact, never sufficient on its own to qualify a pair as a candidate.

A pair qualifies as a candidate only via `same_mark` or `fact_agreement`. Every pair with *any* evidence (qualifying or not) is recorded in `IdentityHypothesis.evidence` — a `fact_conflict`-only pair is fully visible in the trail even though it never becomes a candidate, satisfying "generate every plausible candidate, then score, then reject" and the explainability requirement (considered-and-excluded is distinguishable from never-considered).

One bug caught and fixed during implementation: `mark` was initially swept into the generic shared-aspect `fact_agreement`/`fact_conflict` loop *in addition to* the dedicated `same_mark` check, producing redundant, namespace-unaware duplicate evidence whenever two marks matched. Fixed by excluding `ASPECT_MARK` (and `ASPECT_POSITION`) from that loop — each aspect has exactly one evidence-generation path now.

## Real-data result

Against `SS-GF-01(M).dxf`'s 3 observations: **`N6` and `N7` become mutual candidates** — via `fact_agreement` on `shape=branch`, `orientation=90.0`, `quantity=1`, corroborated by a 1882.5mm centroid distance — while **`N4` gets zero candidates** against both (its facts conflict with everything: different shape, different orientation, ~30,000mm away). This is the Hypothesis Generator independently rediscovering, from non-spatial evidence alone, the same "`N6`/`N7` belong to one upstand detail" relationship the research report found by rendering and visually inspecting the geometry (`docs/research/phase12_cross_view_fusion_research.md` Step 1). It did this without ever computing spatial distance as a qualifying signal — the qualifying evidence was shape/orientation/quantity agreement; distance only corroborated afterward. That's the ordering working as designed, not by coincidence.

`N4`'s hypothesis correctly has 0 candidates *and* a non-empty evidence list (10 items, all `fact_conflict`/`spatial_distance`) — proof the generator considered and rejected pairings with evidence on record, rather than never looking.

## The synthetic acceptance criterion

The original Phase 12.0 acceptance criteria (plan + section + schedule, all marked `N7`, should be grouped) is tested directly — three synthetic observations, no real second machine-readable drawing needed yet, since `same_mark` evidence alone is sufficient to qualify pairs regardless of `drawing_role` classification. All three become mutual candidates of one another, and all three hypotheses remain `status='PENDING'` — proving the shape works correctly ahead of real cross-drawing fixture data (still gated on DWG support, per Phase 12.1's real-data constraint note).
