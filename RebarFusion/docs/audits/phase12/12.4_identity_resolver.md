# Phase 12.4 — Identity Resolver

**Boundary, as specified before this subphase started: Phase 12.4 is allowed to create `PhysicalIdentity` objects, but it is not allowed to reconstruct geometry or modify the reconstruction engine. Its sole responsibility is to make explainable, deterministic decisions about which hypotheses become physical identities, preserving every acceptance and rejection with full provenance.**

## What was built

- `core/fusion/models.py` — `ResolutionDecision` (`hypothesis_uuid`, `anchor_observation`, `candidate_observation`, `outcome`, `rationale`, `overall_confidence`, `promoted`), `Claim` (the object sketched in the research report's Addendum 2, implemented for real here for the first time), and `PhysicalIdentity` — deliberately minimal per review: `uuid`, `claims`, `observations`, `decision_uuid`, `provenance`. No confidence, no scores, no evidence directly on it.
- `core/fusion/identity_resolver.py` — `resolve_identities(scored_hypotheses, observations) -> (decisions, identities)`.
- `run_phase12.py` extended to write `debug/<project>/decisions.json` and `identities.json`.
- `tests/test_phase12_identity_resolver.py` — 6 checks, including an AST-based boundary guardrail (not a substring check) verifying `identity_resolver.py` imports nothing from `core.reconstruction`. All pass.

## The decision rule, and why it isn't one tuned float

A single threshold on `EvidenceScore.overall` (e.g. "accept above 0.7") would be calibrated against exactly one real data point (`N6`<->`N7` at `0.75`) — not a threshold, a guess dressed as one. Instead, the rule reasons over the *shape* of `category_scores`, not a cutoff on the aggregate:

- every scored category `== 1.0` (no contradicting evidence anywhere) → **ACCEPTED**
- any scored category `== 0.0` (a category that's pure contradiction) → **REJECTED**
- otherwise (some support, some contradiction, nothing unanimous) → **REVIEW**

This is boolean over already-computed, already-explained numbers — no new magic constant introduced. It is explicitly labeled "v1" in the module docstring, to be revisited once more real cross-drawing data exists to calibrate against.

## Real-data result: `N6`<->`N7` → REVIEW, zero identities

Under this rule, the one real candidate pair this project can currently test against resolves to **REVIEW**, not ACCEPTED — `category_scores={'fact': 0.75}` is neither unanimous support nor pure contradiction; there's a genuine, real conflict (the two families' `length` facts disagree) alongside genuine agreement (shape/orientation/quantity). **Zero `PhysicalIdentity` objects exist from real data today.** This is the correct, honest outcome, not a shortfall: the project has exactly one machine-readable drawing, so there has never been a real pair of observations with unanimous, conflict-free, independent evidence to accept. The REVIEW decision is preserved in full — 5 rationale items, all of Phase 12.3's evidence for this pair — precisely so a human reviewer (or a future recalibration) can inspect why it wasn't automatically accepted, rather than the pipeline silently doing nothing.

## Synthetic proof the mechanism works

Two checks exercise what real data can't yet:
- **`check_synthetic_accepted`** — a hand-built pair with unanimous, conflict-free evidence (same mark, same shape, agreeing diameter, zero contradictions) resolves to ACCEPTED and produces exactly one `PhysicalIdentity` with both observations as members, one `Claim` per observation (facts copied verbatim, per the "no new inference" rule), `decision_uuid` pointing at the accepting decision, and that decision's UUID present in `provenance`.
- **`check_transitive_clustering`** — A↔B accepted, B↔C accepted, A↔C *never even generated as a candidate pair* (different marks, no shared facts) — still merges into **one** three-observation identity via union-find connected components, not two separate two-observation identities. This is the concrete proof that identity formation is transitive through a bridging observation, matching how a real plan/section/schedule triple would actually resolve (each pair independently evidenced, never all three directly compared against each other).

## `PhysicalIdentity` stayed as small as specified

No field on `PhysicalIdentity` duplicates something a `ResolutionDecision` already owns. `claims` carries only facts (`Claim.facts`, copied from the source observation, no new inference), `observations` is the flat membership list, `decision_uuid` points at the one decision that primarily formed it (the lowest-UUID accepted decision touching only this identity's members, for a deterministic pick when multiple accepted pairs contributed), and `provenance` lists **every** decision — accepted, rejected, or under review — that touched any member observation, so "why wasn't a fourth observation included" stays answerable by reading the identity itself, without a separate query.

## What was explicitly not done

- No geometry, no centerline, no mesh — verified by the AST-based import check, not just a claim.
- No modification of `core/reconstruction/*` or `core/full_pipeline.py`.
- No new evidence categories or generators added to 12.2/12.3 (per your explicit instruction to freeze 12.3 exactly where it is).
- No threshold tuning beyond the boolean rule above — the rule is intentionally conservative given `n=1` real calibration point.
