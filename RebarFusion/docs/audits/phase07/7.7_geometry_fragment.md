# Phase 7.7 — Geometry Fragment Audit

Scope: resolve the "5.34mm artifact" question that Phase 7.6, 9.1 (indirectly, via the N4 seed), 9.2, and 9.3 all independently converged on. 70 components currently sit in Phase 7.6's `review`/`reject` buckets (31 reject, 39 review) across `straight_bar`/`l_bar`/`u_bar`/`stirrup`. This audit traces provenance to a root cause rather than treating them as one undifferentiated "tiny component" bucket.

## Method
Clustered all 70 suspect components by `(recognition_label, sorted edge lengths)` — a cheap but effective geometry fingerprint, since components that are the same *thing* drawn repeatedly will have identical edge-length signatures. Then traced provenance (`CanonicalProvenance.source_handle`/`source_block`) for the largest clusters back to the raw DXF, and visually rendered one instance at both drawing-context scale and true-size zoom.

## Result: 32 distinct clusters, but 2 account for 54% of the population
```
19x  l_bar         edges=(0.5, 4.9)mm
19x  straight_bar  edges=(0.5,)mm
32x  straight_bar  32 distinct singleton lengths (1.6mm - 29.2mm), no shared signature
```

## Root cause, fully resolved: the 38-component cluster is one repeated drafting symbol
Traced provenance for all 38 components in the two dominant clusters:
- **All have `source_block: None`** — native DXF entities directly on the S-RBAR layer, not exploded from an INSERT/block reference.
- **DXF handles are tightly sequential in triplets**: for every occurrence, a standalone tiny LINE (handle `N`), then a LINE (handle `N+1`) + ARC (handle `~N+4`) — e.g. `20D` → `20E`+`211`, `21A` → `21B`+`21E`, `227` → `228`+`22B`, repeating 19 times with near-identical handle offsets throughout the file.
- **The dimensions are bit-identical across all 19 occurrences**: the standalone line is always ~0.5mm, the l_bar pair is always a 0.478mm line + a 4.858mm arc.
- **Visually confirmed isolated**: rendered one instance at both 300mm context and 8mm true-size zoom (`debug/phase07_6/fragment_context.png`). The shape sits alone in open space between (but not touching) three nearby double-line bar representations — it does not connect to, terminate, or mark any adjacent bar geometry.

**Classification: CAD drafting/QA symbol.** Not a hook (the drawing's genuine hooks are single ~9.7mm arcs, a completely different, already-recognized population — see Phase 7 recognition results). Not a leader/dimension (wrong layer semantics for those). Not a block/INSERT explosion artifact (`source_block: None` rules this out explicitly). Not a topology/duplicate-edge artifact (unrelated component set to the still-open Phase 6 duplicate-edge items). Most consistent with a small, manually-repeated verification/reference tick that a drafter or drafting tool placed 19 times on the structural layer — three tiny native primitives per occurrence, always the same shape, never touching real reinforcement.

This closes the loop that all four downstream audits were circling: Phase 7.6 correctly flagged it as physically implausible (though the ratio test placed it just short of "reject"), Phase 9.1/9.2/9.3 kept independently rediscovering the same objects because they *are* the same objects — one symbol, not 38 separate ambiguities.

## Remaining 32: genuinely unresolved, not swept in with the resolved cluster
The other 32 suspect `straight_bar` components share **no common signature** — 32 distinct lengths from 1.6mm to 29.2mm, no repeated handle pattern, no shared block. This population does not cluster into a single explanation the way the 38 did. Honest status: **not resolved by this audit**. They may be individually real-but-short geometry, individual drafting noise, or several different small causes mixed together — clustering alone can't distinguish that, and forcing a classification without evidence would repeat the mistake this whole audit chain has been avoiding (declaring something "explained" without proof). Left as `unresolved_singleton` in `debug/phase07_7/fragment_audit.json`, visible for future investigation rather than silently accepted or rejected.

## One follow-up worth doing, not done this round
Given the 19 `l_bar`-cluster members currently sit in Phase 7.6's `review` tier (not `reject`) purely because the ratio-to-median statistical test landed just above the cutoff (documented in `audit_phase07_6_physical_plausibility.md`), and this audit now has independent, non-statistical (provenance + visual) confirmation they're not real bars, there's a reasonable case for adding a new, principled plausibility rule — e.g. an "isolation" check (no other component's geometry within some radius) — rather than retuning the existing ratio threshold to force a match. Not implemented in this round: didn't want to add a new rule under time pressure without checking it doesn't also misfire on the 41 genuine single-arc hooks (which are also relatively isolated from other bars by nature). Flagging as the natural next increment if this matters before Phase 10.

## Verification
- No source code changed this round — audit only, tracing and classification, consistent with the instruction to resolve the question before continuing to Phase 9.4.

## Verdict
54% of the flagged fragment population (38/70) is now conclusively explained: a single repeated CAD drafting symbol, not a recognition bug, not an engineering error, not 38 separate mysteries. The remaining 46% (32/70) is honestly reported as unresolved rather than forced into the same explanation. This is enough to proceed to Phase 9.4 with a clear conscience about what's understood and what isn't — the open item is now precisely scoped (32 heterogeneous singleton fragments) rather than a vague "94 tiny components, unknown cause."
