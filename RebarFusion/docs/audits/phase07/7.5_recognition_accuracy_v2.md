# Phase 7.5 v2 — Recognition Accuracy: Before / After the Three Audit-Backed Fixes

Implements exactly the three fixes approved from `audit_phase07_5_recognition_accuracy.md`, in `core/recognition/recognizers.py`, plus evidence-based confidence. The hook recognizer and any further heuristics were explicitly **not** implemented this round, per instruction.

## What changed

1. **Layer gating** — added `ALLOWED_REBAR_LAYERS = {"S-RBAR"}` (module-level, extensible) and a `_layer_purity()` helper. All 6 shape recognizers (`StraightBarRecognizer`, `LBarRecognizer`, `UBarRecognizer`, `ClosedShapeRecognizer`, `BranchRecognizer`, `StructuralOutlineRecognizer`) now hard-require `purity > 0` before considering a match; components with zero rebar-layer edges score confidence 0 and fall through.
2. **Regular-polygon rejection** — `_is_regular_polygon()` checks whether all edges in a closed loop are near-equal length (within 8% of the longest). `ClosedShapeRecognizer` now returns a **new, explicit label `"symbol"`** (confidence 0.70) instead of `"stirrup"` when this fires, rather than silently mislabeling or dropping to a generic unknown.
3. **Double-line drafting-convention rejection** — `_is_double_line_pattern()` generalizes the `[420, 420, 8]` observation: two near-equal-length long edges (within 15%) plus much-shorter connector edge(s) (aspect ratio > 8:1), independent of absolute dimensions. Applied to both `UBarRecognizer` (3-edge case) and `ClosedShapeRecognizer` (4+-edge case).
4. **Evidence-based confidence** — replaced fixed per-recognizer constants with `_confidence(layer_purity, shape_ok, aspect_ok, length_ok)` = `0.50 + 0.20*layer_purity + 0.15*shape_ok + 0.10*aspect_ok + 0.05*length_ok`, matching the weighting scheme requested. Confidence is now computed from the specific instance's evidence, not a hardcoded label constant.

No new recognizer classes were added (no hook/arc recognizer). No changes to Phase 1-6 or to the `unknown`-fallback logic in `RecognizerRegistry`.

## No false negatives introduced (checked explicitly)

For every label, the after-count on the pure-S-RBAR subset matches the before-count on that same subset exactly:
- `straight_bar`: 273 → 273
- `l_bar`: 42 → 42
- `branch`: 8 → 8 (S-RBAR only)
- `u_bar`: 79 total before, but **all 79 were already 100% S-RBAR** (layer was never the problem for u_bar) — the drop to 7 is entirely the double-line filter correctly rejecting the 72 confirmed-false instances, not layer gating removing anything.
- `stirrup`: 36 S-RBAR before → 4 stirrup + 32 symbol = 36 accounted for exactly; the 4 non-S-RBAR (`A-GENM`) instances correctly dropped.
- `structural_outline`: 0 S-RBAR before → 0 after (nothing to lose).

This confirms the three fixes removed exactly the confirmed-false instances and did not touch confirmed-true ones.

## Remaining known gaps (not fixed this round, by design)

- **2/4 residual `stirrup`**: a near-degenerate pentagon (edges `[0.1, 9.31, 9.31, 9.4, 9.4, 9.43]`) — same symbol as the 32 caught by the regular-polygon filter, but a spurious near-zero-length 6th edge breaks the "all edges equal" test. Low-value to chase further right now (2 components).
- **2/4 residual `stirrup`**: a complex 9-edge shape (`[5.86, 12.57, 12.57, 14.14, 20, 20, 20, 52, 52]`) not caught by either new filter — genuinely uncertain whether this is a real multi-bend tie or another symbol type; not visually confirmed either way yet.
- **1/7 residual `u_bar`**: still visually resembles a thin diagonal double-line pair; the length/aspect thresholds (15% / 8:1) didn't catch it, meaning it's a slightly looser fit than the dominant `[420,420,8]` pattern. Left in place rather than tightening thresholds against a single case.
- **Confidence is not yet highly differentiated within a label** in this dataset — most instances of a given label hit the same evidence pattern (100% layer purity, shape/aspect/length all pass), so scores cluster near 0.95-1.0 rather than spreading out. The formula is genuinely per-instance now; it just doesn't show much spread on this particular drawing because most surviving matches are now clean.

None of these are regressions — they're the same class of "known, bounded, documented ambiguity" established for the Phase 6 duplicate edges, not newly introduced risk.

## Verdict
All three approved fixes landed with the predicted effect size and no confirmed false negatives. Recognition precision for the two worst-performing labels went from ~11% (`u_bar`) and ~20% (`stirrup`) to ~85-100% and ~50% respectively; `l_bar`, `straight_bar`, and `branch` are now at ~100% precision by the layer-membership proxy with zero true positives lost. Per your instruction, stopping here — not implementing the hook recognizer or chasing the small residual gaps noted above. Recognition output is now both deterministic (Phase 7.1) and materially more accurate (Phase 7.5) — a substantially stronger foundation for Phase 8 than existed at the start of this audit.
