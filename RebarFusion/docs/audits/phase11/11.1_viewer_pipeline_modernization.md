# Phase 11.1 ‚Äî Viewer Pipeline Modernization

Implements the 7-step plan exactly as scoped: infrastructure only, no UI redesign, no new panels, no new rendering features. Goal: the viewer displays exactly the same engineering model `run_phase10.py` produces.

## Step 1 ‚Äî Replaced the legacy pipeline
**New `core/full_pipeline.py`**: the single, canonical Phase 1-10 pipeline (`run_pipeline_through_phase9`, `build_reconstruction`, `run_full_pipeline`). Both `run_phase10.py` and `viewer/controllers/project_controller.py` now call this instead of each maintaining their own copy ‚Äî the exact mechanism that let the viewer's copy silently drift out of sync with every fix made this session (documented in `11.0_viewer_audit.md`).

Not named `core/pipeline.py`: that filename is already used by an unrelated, older prototype architecture (`core/engine.py`, `core/context.py`, `core/geometry/parser.py`/`normalizer.py`) ‚Äî confirmed dead relative to the real phased pipeline (no `run_phaseN.py` script imports it) but still referenced by a few other legacy modules, so left untouched rather than overwritten.

`run_phase10.py`'s `_build_pipeline` (87 lines, a near-exact duplicate of the viewer's now-former `_run_legacy_pipeline`) was deleted entirely in favor of calling `core.full_pipeline.run_full_pipeline`. Re-ran after the refactor: **identical output** (same vertex/face counts, same determinism hash as before the change) ‚Äî confirms the extraction was behavior-preserving, not just structurally cleaner.

**Acceptance criteria, verified**: `tests/test_viewer_pipeline.py` (new) confirms viewer (both its live-pipeline and bundle-loading paths) and `core.full_pipeline` called directly produce identical family/assembly/bar/mesh counts, and identical per-bar centerline point counts (proving the viewer's bars carry the same recovered geometry, not a re-synthesized approximation). No duplicated engineering logic remains in the viewer ‚Äî `project_controller.py` dropped from importing 12 `core.*` modules directly to importing exactly one (`core.full_pipeline`).

## Step 2 ‚Äî FamilyRenderer no longer synthesizes geometry
Rewrote `viewer/renderer/family_renderer.py::_member_mesh` to look up the matching `PhysicalBar` (via `(family.uuid, member.uuid)`, matching how `bar_builder.py` sets `PhysicalBar.member_uuid = member.uuid`) and draw `bar.path` ‚Äî the real Phase 10.1 recovered geometry ‚Äî instead of independently re-deriving a straight line from `family.length`/`orientation`. Falls back to a straight line **only** if no `PhysicalBar` exists for that member at all (e.g. a bundle with Phase 9 data but no Phase 10 reconstruction), never when real geometry is available.

The dashed "missing member" placeholder (`_missing_member_mesh`) legitimately still synthesizes a line ‚Äî there is no real geometry for an inferred-but-undetected member, and the dashed styling already signals it's a placeholder, not real data. This is not the bug the audit found; it's correctly distinct from it.

**Verified**: `test_viewer_pipeline.py` confirms all 6 bars in the test dataset carry `len(path) > 2` (real recovered geometry, not a 2-point line) end to end through the viewer's own code path.

## Step 3/4 ‚Äî Phase 7.6 plausibility and Phase 8 leader reconstruction integrated
Both now flow through automatically as a consequence of Step 1 ‚Äî `core.full_pipeline.run_pipeline_through_phase9` calls `evaluate_plausibility()` and `reconstruct_leaders()` internally (the same calls `run_phase8.py`/`run_phase9.py`/`run_phase10.py` make), so there was no separate viewer-side integration to write. Verified directly: loading `test_project` through the viewer's live-pipeline path produces `leader_repo` with 24 reconstructed leaders and `plausibility` with 334 evaluated components ‚Äî previously the viewer had neither (no `evaluate_plausibility` import existed at all, and leaders came from raw `ezdxf` scanning).

Not implemented this round: visually distinguishing rejected/plausibility-flagged fragments in the viewer (gray color + QA badge, as you sketched) ‚Äî that's a rendering *feature*, out of scope for a pipeline-modernization pass per the stated goal. The data (`WorkbenchProject.plausibility`) is now present and available for that to be built on top of.

## Step 5 ‚Äî Bundle versioning enforced
`core/full_pipeline.PIPELINE_VERSION = "10.4"` (bumped whenever `PipelineResult`'s shape or semantics change). `run_phase10.py` now stamps this into `reconstruction_report.json`. `viewer/project_loader.py::load_workbench_bundle` reads it back and raises a new `BundleVersionMismatch` if missing or non-matching ‚Äî `ProjectController.load_project` already wraps the bundle-loading call in a `try/except`, so a version mismatch now correctly and automatically falls through to the live pipeline instead of silently rendering stale data.

**Verified both directions**: loaded a genuine current bundle successfully (`pipeline_version: 10.4` matches); then edited a copy's stamped version to `9.0-stale` and confirmed `load_workbench_bundle` raises `BundleVersionMismatch` and refuses to load it.

**Found and fixed one more bug while verifying this end-to-end** (not one of the 7 planned steps, but surfaced directly by running the code rather than just reading it): `load_workbench_bundle`'s drawing-selection loop picked the *first* drawing in iteration order regardless of whether it had a registered reader (`if candidate.duplicate_of: continue; if candidate.extension.lower() in {...}: drawing = candidate; break`) ‚Äî for `test_project`, this selected `PW-GF-02(R).dwg` (no reader, no real geometry) instead of `SS-GF-01(M).dxf` (the only drawing with actual pipeline data). The family/bar/mesh *counts* still came out numerically correct only because a fallback in `_preferred_output` silently defaulted to the one real data directory anyway ‚Äî but the bundle was labeled with the wrong drawing name, and a project with genuinely multiple valid drawings would have loaded the wrong one's data outright. Fixed by matching `core.full_pipeline`'s exact selection logic (`capabilities.geometry`, not just file extension).

## Step 6 ‚Äî Renderer contract
Not formalized as a strict type-enforced interface this round (e.g. an ABC requiring `draw(repository)`-style signatures) ‚Äî the existing renderers already follow the pattern informally (each consumes a specific `scene.<attribute>` and never calls pipeline functions itself), and `RecognitionRenderer`/`AnnotationRenderer` were confirmed in `11.0` to already be a stub/dead-code respectively, not violators of this contract. Formalizing it further is lower priority than the concrete staleness bugs fixed above; flagged as a Phase 11.2+ cleanup rather than blocking this freeze.

## Step 7 ‚Äî Validation
`tests/test_viewer_pipeline.py` (new) is exactly this: on every run it compares the live pipeline's output, the bundle-loading path's output, and `core.full_pipeline` called directly, failing loudly on any mismatch rather than silently rendering divergent data. This is a permanent regression test, not a one-time check.
