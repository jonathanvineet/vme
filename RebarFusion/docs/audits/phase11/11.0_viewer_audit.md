# Phase 11.0 — Viewer Audit

Scope: `viewer/` (3550 lines — a real, substantial PySide6 desktop application, not a stub). Goal, per instruction: answer "can the existing viewer faithfully display the frozen Phase 10 pipeline?" before writing any new UI code.

**Method note, stated upfront**: this audit combines code-reading (all stages) with a headless smoke-test launch (`QT_QPA_PLATFORM=offscreen`). It does **not** include live interactive verification of picking accuracy, real-time cross-view highlighting, or measured frame rates — those require actually clicking a rendered GUI, which isn't something I can do without visual interaction. Where a finding depends on code inspection only, it's marked verified-by-reading rather than verified-by-observation; that distinction matters for how much to trust each finding below.

## Stage 1/2 — Data Pipeline & Legacy Code Detection: 3 concrete staleness bugs found

`viewer/controllers/project_controller.py::_run_legacy_pipeline` (the fallback path used whenever no precomputed bundle loads successfully — see Stage 3) has the exact same class of staleness bug found and fixed repeatedly across `run_phase8.py`/`run_phase9.py`/`run_phase10.py` earlier in this project, just never propagated here:

1. **Raw ezdxf leader scanning** (lines 130-135): `doc = ezdxf.readfile(...)`, treats every `G-ANNO-TEXT` LINE as its own independent 2-point leader. This is the exact pre-fix code from `audit_phase08_leader_reconstruction.md` — never replaced with `reconstruct_leaders()`. Same consequence: most genuine leader-based annotations won't resolve.
2. **No Phase 7.6 plausibility integration**: `EngineeringAssociationEngine(wp.graph, wp.comp_repo, engine, wp.recognition_cache)` (line 137) omits the `plausibility=` argument entirely — `evaluate_plausibility` isn't even imported. The viewer's association step has no defense against the fragment/symbol contamination Phase 7.6 was built to filter.
3. **Phase 10 geometry recovery bypassed**: `bar_builder.build_for_assembly(assembly)` (line 157) uses the pre-10.1 one-argument call. It won't crash — my Phase 10.1 fallback logic in `_representative_path` degrades gracefully — but it silently means every bar the viewer renders is the **disproven straight-line synthesis**, not the real recovered geometry from 10.1/10.3. The viewer is currently incapable of showing the L-bend/continuous-tube improvements this session spent two full audit rounds building.

None of these are new problems — they're the same integration-gap pattern found in every other phase, just not yet migrated into the viewer's copy of the pipeline.

## Stage 3 — Bundle Audit: confirmed risk, no version gate exists

`viewer/project_loader.py::load_workbench_bundle` is tried **before** the legacy in-process pipeline (`project_controller.py:60`). It reads JSON directly from `debug/` with **no schema/pipeline/bundle version field anywhere** — confirmed by reading the full 252-line file; no `version` key is written or checked. This is exactly the risk you flagged: if a bundle from before this session's fixes still exists on disk, the viewer would silently load Phase-8-vintage (or older) data and render it as if current, with no way to tell. Given this session's `debug/` directories have been regenerated repeatedly by every `run_phaseN.py` invocation, a stale bundle is a live, not hypothetical, risk for whoever runs the viewer next.

**Not implemented this round** (flagging per your "version every bundle" recommendation, deferred to Phase 11.1 alongside the ProjectController fixes, since both touch the same loading path and should be fixed together, not sequentially).

**The `FamilyRenderer` finding is the most important one in this audit.** It renders each family member via its own `_member_mesh(member, family.length, family.orientation, z)` (`family_renderer.py:41,53`) — an independent straight-line synthesis, exactly the pattern Phase 10.1 replaced in `bar_builder.py`. Right now, both `FamilyRenderer` and the (currently-broken) `BarRenderer`/`MeshRenderer` path show straight lines, so no visible inconsistency exists yet. But the moment Stage 1/2's `ProjectController` fix lands, `BarRenderer`/`MeshRenderer` will correctly show real bent/branch geometry while `FamilyRenderer` continues showing straight-line approximations for the *same* bars — a genuine, predictable cross-view contradiction (Stage 7 concern) that doesn't exist today only because everything is uniformly wrong today. Worth fixing in the same pass as the `ProjectController` change, not separately, precisely to avoid shipping that inconsistency even temporarily.

## Verdict
Your prediction holds: this is an integration-staleness problem, not a broken-architecture problem. Concretely:
- **~70% appears sound**: `BarRenderer`/`MeshRenderer`/`BoundingBoxRenderer` are structurally compatible with current models; the selection-cascade design in `scene.py` is real and reasonable; the app boots without crashing.
- **~20% stale**: the 3 `ProjectController` integration gaps (leader reconstruction, plausibility, geometry recovery), the bundle loader's missing version gate, and `FamilyRenderer`'s straight-line synthesis that will become visibly inconsistent once the above is fixed.
- **~10% missing/dead**: `RecognitionRenderer` (self-admitted stub), `AnnotationRenderer`/`AssociationRenderer` (broken import, unreachable).

## Recommended Phase 11.1 fix order
1. `ProjectController._run_legacy_pipeline`: replace raw ezdxf leader scan with `reconstruct_leaders()`, add `plausibility=` to the association engine, pass `graph`/`entity_by_geom_id`/`comp_repo` to `bar_builder.build_for_assembly()` — the same three fixes already proven correct in `run_phase8.py`/`run_phase9.py`/`run_phase10.py`.
2. `FamilyRenderer`: replace its independent straight-line synthesis with the same recovered-path data `BarRenderer` now has access to, so both views agree — fix in the same pass as #1, not after, to avoid a window where the two views visibly disagree.
3. Add bundle/schema/pipeline version fields to `viewer/project_loader.py` and refuse to load a mismatched bundle, per your explicit recommendation.
4. Remove or fix `AnnotationRenderer`/`AssociationRenderer` (either delete the dead code or fix the import and wire it in) — low priority, currently harmless.
5. Only after 1-3 land: interactive verification pass (Stage 5 panels, Stage 6/7 picking/sync, Stage 8 real performance numbers) — this needs a human or an interactive session, not a code audit.
